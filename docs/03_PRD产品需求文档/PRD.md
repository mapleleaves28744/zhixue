# 智学工坊——基于自进化学习智能体与 LLM Wiki 的个性化资源生成学习空间

## PRD 产品需求文档

版本：V1.0  
适用范围：软件杯 A3 项目需求设计、前后端开发拆分、数据库设计、接口设计、Codex 任务拆解  
核心用户：学生、管理员  
说明：本项目以**学生学习端**为核心，不建设完整教师端。教师相关能力仅作为“课程资料维护 / 演示数据维护 / 课程内容管理”能力并入管理员后台，避免项目偏离个性化学习智能体主线。

---

## 1. 项目背景

高校学生在课程学习过程中普遍存在以下问题：

1. 学习资料分散：课程 PPT、教材 PDF、课堂笔记、实验文档、错题、AI 生成解释分散在不同位置，难以形成统一知识空间。
2. 普通 AI 问答缺少长期记忆：传统 AI 问答通常只解决单次问题，不能持续记录学生的学习状态、薄弱知识点、常见错误和解释风格偏好。
3. 个性化学习资源生成不足：大多数系统只能生成通用讲解、通用题目或通用总结，无法结合学生画像、学习历史、错题记录和课程知识结构持续优化。
4. 知识结构缺少沉淀：学生经常问完问题就结束，AI 生成内容没有进入结构化知识库，无法长期复用，也无法形成课程知识图谱。
5. 学习诊断缺少依据：很多学习系统只展示学习时长、答题数量等表层数据，不能明确告诉学生“哪个知识点薄弱、为什么薄弱、下一步该学什么”。

因此，本项目提出“智学工坊”系统：通过大模型、多智能体、RAG、LLM Wiki、自进化学生模型和学习诊断算法，为高校学生构建一个持续成长的个性化学习空间。

## 2. 项目目标

### 2.1 总体目标

构建一个面向高校学生的个性化学习智能体系统，使学生能够上传课程资料、构建个人 LLM Wiki、进行智能问答、生成个性化学习资源、完成练习诊断，并让系统在使用过程中持续理解学生，逐步优化学习推荐策略。

### 2.2 产品目标

1. 支持学生创建课程学习空间。
2. 支持上传课程资料，并自动解析、切片、向量化。
3. 支持基于课程资料和 Wiki 的智能问答。
4. 支持将课程资料、笔记、AI 回答、错题总结沉淀为 LLM Wiki。
5. 支持自动生成个性化讲解、总结、例题、练习题、复习卡片。
6. 支持记录学生学习行为，构建学生画像。
7. 支持根据答题记录和学习行为进行学习诊断。
8. 支持生成个性化学习路径和复习建议。
9. 支持多智能体围绕学习闭环协作。
10. 支持可控自进化：策略更新有记录、可解释、可回滚。
11. 支持管理员进行系统配置、模型配置、用户管理和演示数据维护。

### 2.3 技术目标

1. 前端采用现代 Web 架构，实现清晰、美观、可演示的学生学习端。
2. 后端提供稳定 API 服务。
3. 使用 PostgreSQL 存储业务数据。
4. 使用 pgvector 支持课程资料向量检索。
5. 接入统一大模型接口，支持 OpenAI-compatible API。
6. 通过多智能体编排实现资料解析、Wiki 构建、问答、诊断、推荐等任务协作。
7. 保证系统具备可扩展、可维护、可演示的工程结构。

## 4. 用户痛点

### 4.1 学生痛点

| 编号 | 痛点 | 具体表现 |
|---|---|---|
| P-01 | 资料分散 | PPT、教材、笔记、错题、AI 回答分散，复习时找不到 |
| P-02 | 不知道自己哪里不会 | 学生只能感觉“学不懂”，无法定位具体薄弱知识点 |
| P-03 | AI 回答不够个性化 | AI 不知道学生基础，经常回答过深或过浅 |
| P-04 | 学习内容无法沉淀 | 每次问答结束后，内容没有形成可复用知识库 |
| P-05 | 错题没有长期价值 | 错题只记录答案，不分析错因和关联知识点 |
| P-06 | 复习路径不清晰 | 不知道先补哪个知识点，再学哪个知识点 |
| P-07 | 课程资料太长 | 学生难以快速整理出知识点、重点和例题 |
| P-08 | AI 生成资源质量不稳定 | 生成结果缺少来源、缺少针对性、难以判断可靠性 |

### 4.2 管理员痛点

| 编号 | 痛点 | 具体表现 |
|---|---|---|
| A-01 | 模型配置复杂 | 不同模型 API、Key、Base URL、参数需要统一管理 |
| A-02 | 演示数据难准备 | 比赛答辩需要稳定、可复现的课程数据和学生行为数据 |
| A-03 | Agent 执行不可见 | 多智能体执行过程如果不可视化，难以排查问题 |
| A-04 | Prompt 不易维护 | 多个 Agent 的 Prompt 分散，版本不可控 |
| A-05 | 自进化更新有风险 | 系统策略自动更新如果不可追踪，可能影响演示稳定性 |

## 5. 核心业务场景

### 5.1 场景一：学生上传课程资料并生成 LLM Wiki

1. 学生进入课程空间。
2. 上传课程 PDF、DOCX、Markdown 或 TXT。
3. 系统解析文件内容。
4. 系统自动切分文本片段。
5. 系统提取知识点。
6. 系统生成知识点 Wiki 页面。
7. 系统建立知识点之间的关系。
8. 学生查看 Wiki 页面和来源引用。

输出结果：

- 课程资料记录；
- 文本切片记录；
- 向量索引；
- Wiki 页面；
- Wiki 页面版本；
- 知识点关系。

### 5.2 场景二：学生基于课程 Wiki 进行智能问答

1. 学生在智能问答页面输入问题。
2. 系统识别问题所属课程和知识点。
3. 系统检索课程资料、Wiki 页面和历史学习记录。
4. 问答 Agent 生成带引用的回答。
5. 系统推荐相关知识点和后续学习资源。
6. 学生对回答进行点赞、点踩或追问。
7. 系统记录反馈并更新学生画像。

输出结果：

- 问答记录；
- 引用来源；
- 相关知识点推荐；
- 学生行为记录；
- 画像更新记录。

### 5.3 场景三：系统生成个性化学习资源

1. 学生选择某个知识点。
2. 系统读取学生画像、掌握度、错题记录和 Wiki 页面。
3. 资源生成 Agent 生成个性化讲解、例题、总结或复习卡片。
4. 质量校验 Agent 检查来源、逻辑和内容完整性。
5. 学生保存资源到个人 Wiki。

输出结果：

- 个性化资源；
- 资源来源；
- 资源类型；
- 保存记录；
- 资源质量校验结果。

### 5.4 场景四：学生练习后触发学习诊断

1. 学生选择知识点练习。
2. 练习 Agent 生成题目。
3. 学生提交答案。
4. 批改 Agent 给出评分和错因分析。
5. 诊断 Agent 更新知识点掌握度。
6. 推荐 Agent 生成下一步学习路径。

输出结果：

- 题目记录；
- 答题记录；
- 批改结果；
- 错因标签；
- 知识点掌握度变化；
- 推荐学习路径。

### 5.5 场景五：自进化 Agent 更新学习策略

1. 系统发现学生在某类知识点上反复出错。
2. 自进化 Agent 分析历史记录。
3. 系统生成策略更新建议。
4. 系统记录更新原因、影响范围和版本号。
5. 对低风险策略自动生效。
6. 对高风险策略提交管理员确认。
7. 学生后续学习中使用新策略。

输出结果：

- 策略更新建议；
- 策略版本记录；
- 生效范围；
- 回滚记录；
- 效果评估记录。

## 7. 自进化 Agent 需求

### 7.1 自进化边界

系统中的自进化不允许自动修改系统代码、不允许自动修改数据库结构、不允许自动修改权限规则。  
自进化仅允许作用于以下对象：

1. 学生画像；
2. 知识点掌握度；
3. 学习偏好；
4. 资源生成策略；
5. 问答解释风格；
6. 学习路径推荐策略；
7. Prompt 模板参数；
8. Wiki 页面补全建议。

### 7.2 需求列表

| 编号 | 需求 | 优先级 |
|---|---|---|
| FR-EVO-001 | 系统应记录学生学习行为，包括提问、追问、答题、浏览、收藏、反馈 | P0 |
| FR-EVO-002 | 系统应根据学习行为更新学生画像 | P0 |
| FR-EVO-003 | 系统应根据答题结果更新知识点掌握度 | P0 |
| FR-EVO-004 | 系统应识别学生反复出错的知识点 | P0 |
| FR-EVO-005 | 系统应根据学生反馈调整资源生成风格 | P1 |
| FR-EVO-006 | 系统应生成学习策略更新建议 | P1 |
| FR-EVO-007 | 每次策略更新应记录触发原因 | P1 |
| FR-EVO-008 | 每次策略更新应记录更新前内容和更新后内容 | P1 |
| FR-EVO-009 | 每次策略更新应生成版本号 | P1 |
| FR-EVO-010 | 系统应支持回滚到历史策略版本 | P2 |

| FR-EVO-012 | 系统应展示“为什么系统这样推荐”的解释 | P1 |

### 7.3 自进化策略分级

| 策略类型 | 是否自动生效 | 示例 |
|---|---|---|
| 掌握度更新 | 自动 | 栈的掌握度从 60 调整为 72 |
| 错因标签更新 | 自动 | 新增“递归终止条件不清” |
| 学习偏好更新 | 半自动 | 系统判断学生偏好图示解释 |
| 资源生成风格更新 | 半自动 | 优先生成代码示例 |
| Prompt 模板参数更新 | 管理员确认 | 修改全局讲解模板 |
| 系统代码更新 | 禁止 | Agent 不允许自动改代码 |

## 19. MVP 范围

MVP 目标是完成可演示、可闭环的最小版本。

### 19.1 MVP 必做功能

| 模块 | MVP 功能 |
|---|---|
| 用户系统 | 学生注册、登录、管理员登录 |
| 课程空间 | 创建课程、查看课程 |
| 资料上传 | 上传 PDF、DOCX、TXT、Markdown |
| 文档解析 | 文本解析、切片、向量化 |
| 智能问答 | 基于课程资料 RAG 问答，支持引用 |
| LLM Wiki | 自动生成知识点页面，支持查看 |
| 资源生成 | 生成讲解、总结、例题 |
| 练习模块 | 生成题目、提交答案、批改、错因分析 |
| 学生画像 | 记录掌握度、薄弱点、偏好摘要 |
| 学习诊断 | 生成简单诊断报告 |
| 学习推荐 | 推荐下一步知识点 |
| 自进化 | 记录画像更新和策略建议 |
| 多智能体 | 至少实现任务路由、问答、Wiki、资源生成、诊断 Agent |
| 管理员后台 | 模型配置、Prompt 查看、日志查看 |
| 演示数据 | 一门课程、一个学生账号、完整演示链路 |

### 19.2 MVP 不做功能

| 功能 | 原因 |
|---|---|
| 完整教师端 | 容易偏离学生学习智能体主线 |
| 班级管理 | 非 A3 核心能力 |
| 作业发布与批改 | 工作量大，偏教师管理系统 |
| 复杂 OCR | MVP 优先支持文本型资料 |
| 完整知识图谱推理 | 先做关系展示和简单路径推荐 |
| 自动代码级自进化 | 风险高，不适合比赛系统 |
| 多租户 SaaS | 非比赛重点 |

| 模块 | 增强功能 |
|---|---|
| LLM Wiki | 页面版本回滚、矛盾检测、AI 补全、健康度检查 |
| 知识图谱 | 可视化知识关系、前置知识路径、易混知识点网络 |
| 自进化 | 策略版本管理、效果评估、回滚机制 |
| 多智能体 | Agent 调用链可视化、失败降级、工具权限管理 |
| 学习诊断 | 遗忘风险、学习趋势、错因聚类 |
| 推荐系统 | 个性化学习路径、复习计划、资源推荐理由 |
| 多模态 | 图片题识别、截图解释、公式识别 |
| 管理员后台 | Prompt 版本管理、演示数据一键初始化 |
| 测试评估 | RAG 检索评估、生成质量评估、Agent 稳定性评估 |
| 部署 | Docker Compose 一键部署 |

## 21. 验收标准

### 21.1 功能验收标准

| 编号 | 验收项 | 通过标准 |
|---|---|---|
| AC-001 | 用户登录 | 学生和管理员可正常登录并进入不同页面 |
| AC-002 | 创建课程空间 | 学生可创建课程并进入课程详情页 |
| AC-003 | 上传资料 | 学生可上传文件并看到解析状态 |
| AC-004 | 文档解析 | 系统能提取文本并生成切片 |
| AC-005 | 向量检索 | 系统能根据问题检索相关资料片段 |
| AC-006 | 智能问答 | 系统能基于资料回答问题并返回引用 |
| AC-007 | Wiki 生成 | 系统能生成知识点 Wiki 页面 |
| AC-008 | 资源生成 | 系统能生成讲解、总结、例题 |
| AC-009 | 练习批改 | 系统能生成题目并批改答案 |
| AC-010 | 学习诊断 | 系统能展示薄弱知识点和掌握度 |
| AC-011 | 学习推荐 | 系统能推荐下一步学习内容 |
| AC-012 | 学生画像 | 系统能展示学生画像摘要 |
| AC-013 | 自进化日志 | 系统能记录画像或策略更新原因 |
| AC-014 | Agent 日志 | 系统能记录 Agent 执行过程 |
| AC-015 | 管理员配置 | 管理员能配置模型参数 |

### 21.2 演示验收标准

系统应能完成以下演示链路：

1. 学生登录系统。
2. 创建《数据结构》课程空间。
3. 上传课程资料。
4. 系统解析资料并生成知识点 Wiki。
5. 学生提问“递归为什么和栈有关？”。
6. 系统基于资料和 Wiki 返回带引用回答。
7. 系统推荐相关知识点“函数调用栈”。
8. 学生生成“递归调用栈”个性化讲解。
9. 学生完成相关练习题。
10. 系统批改并识别错因。
11. 系统更新学生画像。
12. 系统生成学习诊断报告。
13. 系统推荐下一步学习路径。
14. 自进化 Agent 生成策略更新建议。
15. 管理员后台查看 Agent 执行日志和模型配置。

### 21.3 非功能验收标准

| 编号 | 验收项 | 通过标准 |
|---|---|---|
| AC-NFR-001 | API 响应 | 普通接口能稳定返回统一 JSON |
| AC-NFR-002 | 流式输出 | 智能问答能逐步输出内容 |
| AC-NFR-003 | 错误处理 | 文件解析失败、模型失败有明确提示 |
| AC-NFR-004 | 权限控制 | 学生不能访问其他学生数据 |
| AC-NFR-005 | 日志记录 | 模型调用、Agent 执行、异常均有日志 |
| AC-NFR-006 | 数据删除 | 学生能删除个人资料和学习记录 |
| AC-NFR-007 | 部署运行 | 系统能在本地或服务器稳定启动 |

## 22. 后续拆分建议

为了便于 Codex 后续拆任务，建议按以下顺序开发：

1. 用户认证与角色权限；
2. 课程空间；
3. 文件上传与解析；
4. 文档切片与向量检索；
5. 智能问答；
6. LLM Wiki 页面生成；
7. 个性化资源生成；
8. 练习与批改；
9. 学生画像；
10. 学习诊断；
11. 学习路径推荐；
12. Agent 执行日志；
13. 自进化策略记录；
14. 管理员后台；
15. 演示数据与答辩优化。

MVP 阶段必须优先保证一条主线跑通：

> 资料上传 → 知识切片 → Wiki 生成 → 智能问答 → 练习诊断 → 学生画像更新 → 个性化推荐。
