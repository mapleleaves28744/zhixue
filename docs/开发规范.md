# 智学工坊开发规范

## 总则

- 每次只完成当前任务，不扩大重构范围。
- 前后端实现必须与 `docs/` 设计保持一致。
- 不提交 `.env`、API Key、真实用户隐私数据。
- 数据库结构变化必须通过 Alembic migration。
- 大模型调用必须通过统一 LLM Provider，缺少真实 Key 时使用 Mock Provider。
- 当前阶段采用“本地优先，Docker 后置”：业务功能先在本地 Python/Node/PostgreSQL/Redis 环境跑通，Docker Compose 作为后续部署阶段验收，不作为日常开发第一阻塞项。
- 数据库和缓存也采用本机优先：默认连接 `localhost:5432` 的 PostgreSQL 和 `localhost:6379` 的 Redis，不在非部署任务中默认依赖 Docker 数据库。

## 前端规范

- 使用 Next.js App Router、TypeScript、Tailwind CSS 和 shadcn/ui 源码组件。
- 页面只做组合和页面级状态，复杂请求放入 `frontend/services/`。
- 通用组件放入 `frontend/components/`，类型放入 `frontend/types/`。
- 避免 `any`，API 响应必须使用泛型类型。
- 视觉遵守 Luminous Warmth：暖白背景、金棕主色、暖橙强调、玻璃拟态卡片、轻阴影和清晰层级。
- 业务页面不得散落裸 `fetch`，统一使用 `frontend/lib/request.ts`。

### Stitch 静态页接入规范

当前学生端以 Stitch 静态原型为视觉基准。后续前端开发必须遵守：

- 默认保留 `frontend/public/stitch-pages/*.html` 的视觉结构和导航，不重做页面。
- `frontend/app/*/page.tsx` 顶层路由默认继续使用 `StitchFrame` 加载对应静态页。
- 后端能力完成后，必须在对应 Stitch 静态页中接入 API，让用户可通过前端使用。
- 静态页公共请求、鉴权、toast、课程 ID 解析等逻辑放在 `frontend/public/stitch-pages/zhixue-static-api.js`。
- 单页交互脚本可以写在对应 `.html` 底部，但应保持轻量；复杂公共逻辑抽到 `zhixue-static-api.js`。
- 静态页脚本调用后端必须经过 `/api/v1`，不得绕过后端直接访问数据库或大模型。
- 原静态占位数据不得冒充真实结果；接入 API 后应显示真实数据、真实空状态或明确的 Mock/未生成状态。
- 不得随意修改已设计好的导航栏、侧边栏、顶部栏、主色、卡片结构和主要布局。
- 只有任务明确要求并经人工确认时，才允许将某个 Stitch 模块正式 React 组件化。

当前功能默认接入位置：

| 功能 | 默认页面 |
|---|---|
| 课程 CRUD | `/courses` |
| 资料上传、解析、切片、向量化、知识点抽取、Wiki 生成 | `/knowledge` |
| Wiki、知识图谱、RAG 检索 | `/knowledge` |
| AI Tutor、引用、资源生成 | `/assistant` |
| 练习、错题、诊断 | `/practice` |
| 学习总览、推荐、任务 | `/dashboard` |
| 画像、长期记忆、学习路径、自进化、Agent 日志 | `/path-profile` |

## 后端规范

- API Router 只负责参数接收、依赖注入、调用 Service 和返回响应。
- Service 层负责业务流程、权限上下文、事务和 Agent/LLM 调用。
- Repository 层只处理数据库查询和持久化，不写业务决策。
- Python 代码必须使用类型注解。
- 业务异常使用统一异常体系，不抛裸字符串异常。

## API 与响应

- 业务接口统一挂载 `/api/v1`。
- 成功响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "request_id": "req_xxx"
}
```

- 错误响应必须包含 `code`、`message`、`detail`、`request_id`。
- 涉及 `user_id` 和 `course_id` 的查询必须校验当前用户权限。

## 数据库规范

- 表名使用 snake_case 复数，如 `users`、`courses`。
- SQLAlchemy Model、Pydantic Schema 和 migration 必须同步。
- pgvector 检索必须带 `course_id` 过滤，避免跨课程或跨学生泄露。
- 本地开发 `.env` 的 `DATABASE_URL` 应指向本机 PostgreSQL；Docker 内部主机名 `postgres` 只用于 Docker Compose 部署环境。

## Agent、Wiki 与自进化

- Agent 类继承 `BaseAgent`，输入输出使用 `AgentContext` 和 `AgentResult`。
- Agent 不允许自动修改源代码、数据库结构、权限规则或部署配置。
- Wiki 内容应保留来源、版本和关系。
- 自进化策略必须可解释、可记录、可版本化、可回滚、有证据和风险等级。

## Prompt 规范

- Prompt 不散落在业务代码中，优先放入 `backend/app/prompts/` 或 `prompt_versions`。
- Prompt 按场景命名，如 `tutor.qa`、`wiki.generate`、`resource.generate`、`evolution.analyze`。

## 测试与提交

- 前端任务至少运行 `npm run typecheck` 和 `npm run build`。
- 后端任务至少运行相关 pytest 或启动 FastAPI 检查。
- 数据库任务运行 `alembic upgrade head`。
- 非 Docker 任务默认优先本地验证；只有 Dockerfile、docker-compose 或部署专项任务才强制运行 Docker 验证。
- 如果 Docker 与本地运行冲突，先保证本地主链路可用，并把 Docker 问题记录为后续部署 TODO。
- 每个阶段完成后必须对照 `docs/19_测试方案/12_阶段运行验收清单.md` 做本地验收；也可以运行 `scripts/local_check.ps1` 执行基础检查。
- 输出中必须说明修改文件、数据库变更、API 变更、测试结果和风险。
