# 1. 自进化系统的定义

## 1.1 定义

“自进化学习智能体系统”是智学工坊中的个性化学习策略优化系统。它通过持续收集学生在学习过程中的提问、答题、浏览、反馈、收藏、复习等行为数据，分析学生的知识掌握状态、学习偏好、薄弱知识点、常见错误模式和适合的解释方式，并在受控范围内更新学生长期画像、学习记忆、资源生成策略、答疑风格、学习路径推荐策略、复习策略和 Wiki 知识组织结构。

本项目中的“自进化”不是让 Agent 自动修改代码，也不是让智能体无限制改变系统规则，而是指：

> 系统基于学生真实学习数据，对学习相关的个性化策略进行可解释、可记录、可版本化、可回滚的受控更新。

## 1.2 自进化的核心边界

允许更新：

1. 学生长期画像；
2. 学生学习偏好；
3. 学生薄弱知识点；
4. 学生常见错误模式；
5. 学生适合的解释风格；
6. 个性化提示词模板参数；
7. 学习策略版本；
8. 推荐策略记录；
9. Wiki 知识组织结构建议。

不允许更新：

1. 系统源代码；
2. 数据库表结构；
3. 用户权限规则；
4. 安全策略；
5. 生产环境配置；
6. 第三方模型接口密钥；
7. 管理员账号权限；
8. 未经确认的全局 Prompt 模板。

## 1.3 设计原则

| 原则 | 说明 |
|---|---|
| 数据驱动 | 所有更新都必须来自学习行为、答题结果、反馈记录或 Wiki 使用记录 |
| 可解释 | 每次更新都要记录触发原因、依据数据和影响范围 |
| 可版本化 | 策略、画像、提示词参数、推荐策略都要保留版本记录 |
| 可回滚 | 更新后效果变差或误判时可以恢复到历史版本 |
| 最小影响 | 每次更新优先只影响单个学生，不直接影响全局策略 |
| 人工可控 | 中高风险更新需要人工确认或管理员确认 |
| 可展示 | 前端需要展示系统如何“越来越懂学生” |

# 2. 为什么它适合本赛题

## 2.1 契合“个性化资源生成”

A3 赛题关注基于大模型的个性化学习资源生成。自进化机制能够让资源生成不再是一次性生成，而是随着学生学习历史不断优化。

例如：

| 学生表现 | 系统进化结果 |
|---|---|
| 多次答错递归题 | 后续递归讲解优先加入调用栈图示 |
| 多次点踩长篇回答 | 后续答疑默认更简洁 |
| 收藏代码示例较多 | 后续资源生成增加代码示例 |
| 经常混淆栈和队列 | 后续资源加入对比表和易错提醒 |
| 复习任务经常逾期 | 调整复习提醒频率和任务粒度 |

## 2.2 契合“学习多智能体系统”

自进化不是单个模块完成，而是多个智能体围绕学习闭环协作：

| 智能体 | 作用 |
|---|---|
| 行为采集 Agent | 收集提问、答题、浏览、反馈等学习行为 |
| 学习诊断 Agent | 分析掌握度、薄弱点、错因模式 |
| 学生画像 Agent | 更新学生长期画像和学习偏好 |
| 资源生成 Agent | 根据画像生成个性化讲解、题目、总结 |
| 推荐 Agent | 生成学习路径和复习计划 |
| 自进化 Agent | 判断是否需要生成新策略版本 |
| 质量校验 Agent | 检查策略更新是否合理、是否依据充分 |

## 2.3 契合高校学习场景

高校课程具有知识点多、前后依赖强、学生基础差异明显的特点。自进化机制可以帮助系统逐步识别：

1. 学生在哪些知识点上长期薄弱；
2. 学生更适合哪种解释风格；
3. 学生常见错误是否集中在概念、公式、代码实现或迁移应用；
4. 学生当前最需要补充哪些前置知识；
5. 学生的个人 Wiki 哪些页面需要补全或重组；
6. 学生后续应该优先学习什么内容。

## 2.4 比赛展示价值

自进化机制具备明显展示效果：

1. 展示系统第一次对学生的默认回答；
2. 展示学生多次练习后暴露的薄弱知识点；
3. 展示系统自动更新学生画像；
4. 展示系统生成新的学习策略版本；
5. 展示后续资源生成方式发生变化；
6. 展示推荐路径根据学生状态重新排序；
7. 展示策略版本记录和回滚能力。

# 3. 自进化闭环设计

自进化闭环包含 8 个阶段：

```text
收集学习行为
  → 分析学生状态
  → 更新学生画像
  → 更新学习记忆
  → 优化资源生成策略
  → 优化推荐策略
  → 生成新版本策略
  → 记录版本和效果
```

## 3.1 收集学习行为

系统需要记录学生在学习端产生的关键行为事件。

### 行为类型

| 行为类型 | 示例 | 用途 |
|---|---|---|
| 提问行为 | 学生问“递归为什么和栈有关” | 判断疑惑知识点 |
| 追问行为 | 学生连续追问同一概念 | 判断理解困难程度 |
| 答题行为 | 学生提交练习答案 | 计算知识点掌握度 |
| 错题行为 | 学生答错某类题 | 分析常见错误模式 |
| 浏览行为 | 学生查看某个 Wiki 页面 | 判断关注知识点 |
| 收藏行为 | 学生收藏某类学习资源 | 判断资源偏好 |
| 反馈行为 | 点赞、点踩、标记无用 | 判断生成内容有效性 |
| 复习行为 | 完成或跳过复习任务 | 判断复习策略有效性 |

### 行为事件字段建议

```json
{
  "event_id": "evt_001",
  "user_id": "u_001",
  "course_id": "c_001",
  "knowledge_id": "k_stack_recursion",
  "event_type": "exercise_wrong",
  "event_content": {
    "question_id": "q_001",
    "answer_result": "wrong",
    "error_tag": "递归终止条件不清"
  },
  "created_at": "2026-05-20T10:00:00"
}
```

## 3.2 分析学生状态

学习诊断 Agent 根据行为事件分析学生状态。

分析内容包括：

1. 当前薄弱知识点；
2. 知识点掌握度变化；
3. 常见错误模式；
4. 学习偏好变化；
5. 资源反馈趋势；
6. 复习完成情况；
7. 当前学习路径执行情况；
8. Wiki 页面使用情况。

## 3.3 更新学生画像

学生画像需要动态更新，但不能随意覆盖。每次画像更新都要记录依据。

画像字段包括：

| 画像维度 | 示例 |
|---|---|
| 知识掌握度 | 栈：78，递归：45，二叉树遍历：62 |
| 薄弱知识点 | 递归调用栈、循环队列判满条件 |
| 常见错误模式 | 概念混淆、边界条件遗漏 |
| 解释风格偏好 | 图示解释、代码示例、分步骤讲解 |
| 资源偏好 | 例题解析、对比表、复习卡片 |
| 学习节奏 | 偏好短任务、晚间学习活跃 |
| 学习目标 | 考试复习、实验实现、课程补弱 |

## 3.4 更新学习记忆

长期学习记忆只保存对个性化有长期价值的信息，不保存所有聊天内容。

示例：

```text
学生在“递归调用栈”相关问题中连续出现理解困难。
学生更容易接受“图示 + 代码示例”的解释方式。
学生在“二叉树遍历”题目中经常混淆前序遍历和中序遍历。
学生对长篇概念解释反馈较差，更偏好短步骤解释。
```

## 3.5 优化资源生成策略

资源生成策略根据学生状态调整。

| 学生状态 | 资源生成策略 |
|---|---|
| 基础薄弱 | 先补前置知识，再讲当前知识点 |
| 喜欢代码示例 | 增加代码片段和执行过程说明 |
| 常犯概念混淆错误 | 增加概念对比表和易混点提醒 |
| 即将考试 | 生成重点总结、易错题和速记卡片 |
| 练习正确率低 | 生成基础题和逐步引导题 |
| 追问次数高 | 回答增加分步骤解释和类比 |

## 3.6 优化推荐策略

推荐策略根据掌握度、知识图谱、学习目标和复习情况调整。

| 情况 | 推荐策略 |
|---|---|
| 当前知识点掌握度低 | 推荐前置知识和基础练习 |
| 错题集中在某类知识点 | 推荐针对性补弱路径 |
| 复习任务逾期 | 推荐更短的复习任务 |
| 学生即将考试 | 推荐高频考点和错题回顾 |
| Wiki 页面多次浏览 | 推荐相关练习和总结资源 |

## 3.7 生成新版本策略

当系统判断学生状态发生稳定变化时，生成新的策略版本。

策略版本包括：

1. 策略类型；
2. 适用用户；
3. 适用课程；
4. 适用知识点；
5. 更新前策略；
6. 更新后策略；
7. 触发原因；
8. 证据数据；
9. 风险等级；
10. 生效状态；
11. 创建时间；
12. 回滚目标版本。

## 3.8 记录版本和效果

每次策略生效后，系统需要观察效果。

观察指标包括：

1. 后续答题正确率是否提升；
2. 后续追问次数是否减少；
3. 资源反馈是否变好；
4. 学习路径完成率是否提升；
5. 复习任务完成率是否提升；
6. 学生是否继续点踩相同类型回答。

# 5. 不允许自动进化的对象

以下对象不允许 Agent 自动更新：

| 对象 | 禁止原因 |
|---|---|
| 系统源代码 | 存在安全和稳定性风险 |
| 数据库表结构 | 可能导致数据损坏 |
| 用户权限规则 | 可能造成越权 |
| 管理员账号 | 涉及系统控制权 |
| API Key 和密钥 | 涉及安全风险 |
| 部署配置 | 可能导致系统不可用 |
| 全局安全策略 | 必须人工审核 |
| 全局 Prompt 原文 | 可能影响所有用户 |
| 用户原始上传文件 | 应保持原始资料不被篡改 |
| 历史答题记录 | 应保持学习证据不可篡改 |
| 历史行为日志 | 应保持审计可信 |

# 7. 触发条件

自进化不应每次行为都立即触发完整分析，否则会造成成本高、结果不稳定。建议采用规则触发 + 定时触发 + 手动触发三类机制。

## 7.1 规则触发

| 触发条件 | 触发动作 |
|---|---|
| 同一知识点连续 3 次答错 | 触发薄弱知识点分析 |
| 同类错因出现 3 次以上 | 触发常见错误模式更新 |
| 同一问题连续追问 2 次以上 | 触发解释风格分析 |
| 某类回答连续被点踩 2 次 | 触发答疑风格调整建议 |
| 学生连续收藏同类资源 | 触发资源偏好更新 |
| 复习任务连续 2 次未完成 | 触发复习策略调整 |
| 某 Wiki 页面被频繁访问但练习错误率高 | 触发 Wiki 页面补全建议 |

## 7.2 定时触发

| 周期 | 任务 |
|---|---|
| 每日 | 汇总当天学习行为，更新轻量画像 |
| 每周 | 生成学习诊断报告和策略建议 |
| 每月 | 更新长期学习记忆和稳定偏好 |

## 7.3 手动触发

| 用户 | 可触发动作 |
|---|---|
| 学生 | 重新生成学习诊断、刷新推荐路径、清空学习记忆 |
| 管理员 | 重新计算画像、重建演示数据、回滚策略版本 |

# 8. 版本管理机制

## 8.1 版本对象

需要版本化管理的对象包括：

1. 学生画像版本；
2. 学习策略版本；
3. 推荐策略版本；
4. 个性化 Prompt 参数版本；
5. Wiki 页面结构版本；
6. 资源生成策略版本。

## 8.2 版本字段

```json
{
  "version_id": "sv_001",
  "user_id": "u_001",
  "course_id": "c_001",
  "version_type": "learning_strategy",
  "version_no": 3,
  "previous_version_id": "sv_000",
  "change_summary": "递归相关讲解改为优先使用调用栈图示和分步骤解释",
  "before_snapshot": {},
  "after_snapshot": {},
  "evidence": [
    "递归题连续答错 3 次",
    "递归相关问答追问 4 次",
    "学生收藏调用栈图示讲解"
  ],
  "risk_level": "low",
  "status": "active",
  "created_at": "2026-05-20T10:00:00"
}
```

## 8.3 风险等级

| 风险等级 | 说明 | 生效方式 |
|---|---|---|
| low | 只影响单个学生的局部策略 | 自动生效 |
| medium | 影响学生多个模块，如问答和推荐 | 学生确认或系统提示后生效 |
| high | 影响全局模板、管理员配置或大量数据 | 管理员确认后生效 |

## 8.4 版本状态

| 状态 | 说明 |
|---|---|
| draft | 草稿，未生效 |
| pending | 等待确认 |
| active | 当前生效 |
| rollbacked | 已回滚 |
| rejected | 已拒绝 |
| expired | 已过期 |

# 9. 回滚机制

## 9.1 回滚对象

支持回滚：

1. 学习策略版本；
2. 推荐策略版本；
3. Prompt 参数版本；
4. Wiki 页面结构建议；
5. 学生画像摘要；
6. 长期记忆摘要。

不支持回滚：

1. 原始学习行为日志；
2. 原始答题记录；
3. 原始上传资料；
4. 系统安全配置。

## 9.2 回滚流程

```text
用户或管理员选择历史版本
  → 系统读取当前版本和目标版本
  → 校验目标版本是否可恢复
  → 生成回滚记录
  → 将目标版本设为 active
  → 当前版本设为 rollbacked
  → 写入审计日志
  → 后续问答、推荐、资源生成使用回滚后的策略
```

## 9.3 回滚原因

常见回滚原因：

1. 策略更新后答题正确率下降；
2. 学生反馈生成内容不适合；
3. 推荐路径不合理；
4. 系统误判学习偏好；
5. Wiki 结构调整导致页面混乱；
6. 演示环境需要恢复稳定版本。

# 15. 典型场景

## 15.1 场景一：系统发现学生递归薄弱

### 初始状态

学生在《数据结构》课程中学习“递归调用栈”。

### 行为

1. 学生提问：“递归为什么和栈有关？”
2. 学生追问：“为什么函数没有执行完就保存起来？”
3. 学生做 3 道递归题，答错 2 道。
4. 学生点踩了一次纯文字解释。
5. 学生收藏了一篇调用栈图示讲解。

### 系统分析

系统判断：

1. 学生薄弱知识点：递归调用栈；
2. 常见错误：无法理解函数调用返回过程；
3. 解释偏好：图示 + 分步骤；
4. 推荐策略：先复习栈，再学习递归调用栈。

### 策略更新

生成策略版本：

```text
策略类型：答疑风格策略
更新内容：递归相关知识点优先采用“图示描述 + 分步骤执行 + 代码示例”
触发依据：连续错题、追问记录、收藏行为、点踩反馈
风险等级：low
状态：active
```

### 展示效果

学生再次提问递归问题时，系统回答结构变为：

1. 先说明栈的后进先出；
2. 再解释函数调用栈；
3. 给出代码示例；
4. 按调用顺序和返回顺序分步骤解释；
5. 推荐练习题。

## 15.2 场景二：系统优化复习策略

### 行为

1. 学生连续两次未完成复习任务；
2. 学生每次只完成 5 分钟以内的短任务；
3. 学生对长篇复习资料点击率低。

### 系统分析

系统判断：

1. 学生不适合长复习任务；
2. 学生适合短卡片式复习；
3. 复习任务应拆成更小粒度。

### 策略更新

```text
复习策略从“每天复习 3 个知识点，每个知识点 15 分钟”
调整为“每天复习 5 张知识卡片，每张 3 分钟”
```

## 15.3 场景三：系统优化 Wiki 组织结构

### 行为

1. 学生频繁查看“栈”和“递归”页面；
2. 学生在“递归调用栈”相关题目上错误率高；

# 16. 伪代码流程

## 16.1 行为记录后触发自进化判断

```python
def record_behavior_event(user_id, course_id, event_type, payload):
    event = BehaviorEvent.create(
        user_id=user_id,
        course_id=course_id,
        event_type=event_type,
        event_payload=payload
    )

    trigger = EvolutionTriggerService.check(event)

    if trigger.should_trigger:
        EvolutionService.run_analysis(
            user_id=user_id,
            course_id=course_id,
            trigger_type=trigger.trigger_type,
            trigger_event_id=event.id
        )

    return event
```

## 16.2 自进化分析流程

```python
def run_evolution_analysis(user_id, course_id, trigger_type):
    behavior_summary = BehaviorService.get_recent_summary(user_id, course_id)
    profile = ProfileService.get_profile(user_id, course_id)
    mastery = MasteryService.get_mastery_snapshot(user_id, course_id)
    memories = MemoryService.get_active_memories(user_id, course_id)

    diagnosis = DiagnosisAgent.analyze(
        behavior_summary=behavior_summary,
        profile=profile,
        mastery=mastery
    )

    strategy_suggestion = EvolutionAgent.generate_strategy(
        diagnosis=diagnosis,
        profile=profile,
        memories=memories
    )

    risk_level = RiskService.evaluate(strategy_suggestion)

    version = StrategyService.create_version(
        user_id=user_id,
        course_id=course_id,
        strategy_suggestion=strategy_suggestion,
        risk_level=risk_level
    )

    if risk_level == "low":
        StrategyService.activate(version.id)
    else:
        StrategyService.mark_pending(version.id)

    EvolutionLogService.record(
        user_id=user_id,
        course_id=course_id,
        trigger_type=trigger_type,
        diagnosis=diagnosis,
        strategy_version_id=version.id
    )

    return version
```

## 16.4 策略效果评估

```python
def evaluate_strategy_effect(strategy_version_id):
    strategy = StrategyService.get(strategy_version_id)

    before_metrics = MetricsService.get_before_metrics(strategy)
    after_metrics = MetricsService.get_after_metrics(strategy)

    result = compare_metrics(before_metrics, after_metrics)

    StrategyEffectMetric.create(
        strategy_version_id=strategy_version_id,
        metric_name="answer_accuracy",
        before_value=before_metrics.accuracy,
        after_value=after_metrics.accuracy,
        evaluation_result=result
    )

    if result == "worse":
        StrategyService.suggest_rollback(strategy_version_id)

    return result
```

# 17. MVP 实现方案

MVP 的目标不是实现复杂智能体自治，而是实现一个可以演示的自进化闭环。

## 17.1 MVP 必做能力

| 能力 | MVP 实现方式 |
|---|---|
| 行为采集 | 记录提问、答题、错题、反馈、收藏 |
| 学生画像 | 使用 JSON 字段存储画像摘要、偏好、薄弱点 |
| 掌握度更新 | 根据答题正确率和错题次数用规则计算 |
| 错因分析 | 由题目标签和大模型简要分析生成 |
| 策略生成 | 根据规则 + LLM 生成策略建议 |
| 策略版本 | 保存 before / after / evidence |
| 低风险自动生效 | 单学生局部策略自动生效 |
| 回滚 | 支持恢复到上一版本 |
| 前端展示 | 展示画像、策略时间线、推荐理由 |

## 17.2 MVP 规则示例

### 掌握度计算

```text
初始掌握度：60
答对一次：+8
答错一次：-10
查看讲解：+2
完成复习：+5
连续答错：额外 -5
掌握度限制在 0-100
```

### 薄弱点判定

```text
掌握度 < 60：标记为薄弱
同一知识点最近 5 次练习错误率 > 50%：标记为薄弱
同一知识点追问次数 >= 3：标记为理解困难
```

### 解释风格更新

```text
学生点踩长篇回答 >= 2 次：answer_length = short
学生收藏代码资源 >= 3 次：include_code_example = true
学生在概念题错误多：include_misconception_warning = true
学生追问次数多：include_step_by_step = true
```

## 17.3 MVP 展示页面

至少实现 3 个展示区域：

1. 学生画像卡片；
2. 自进化策略时间线；
3. 推荐理由卡片。

## 17.4 MVP 演示链路

```text
学生学习“递归调用栈”
  → 连续答错 3 道题
  → 系统识别“递归调用栈”为薄弱知识点
  → 系统更新学生画像
  → 系统生成新策略：递归讲解优先使用图示 + 分步骤
  → 学生再次提问递归问题
  → 回答风格发生变化
  → 前端展示策略更新记录和推荐理由
```

# 18. 增强版实现方案

增强版用于冲奖展示，重点增强可解释性、可视化和策略效果评估。

## 18.1 增强能力

| 能力 | 增强实现 |
|---|---|
| 策略效果评估 | 对比策略更新前后的正确率、追问次数、反馈评分 |
| Wiki 结构进化 | 自动发现页面缺失、重复和关系缺失 |
| 多 Agent 可视化 | 展示诊断 Agent、自进化 Agent、推荐 Agent 的调用链 |
| 风险分级 | 中高风险策略进入 pending 状态 |
| 策略审核 | 管理员可确认、拒绝、回滚策略 |
| 记忆合并 | 合并相似长期记忆，降低重复记忆 |
| 画像解释 | 展示每个画像标签的证据来源 |
| 推荐解释 | 展示推荐路径的知识图谱依据 |
| 自动 AB 对比 | 对不同资源生成策略做效果比较 |

## 18.2 增强版前端展示

新增页面：

1. 自进化中心；
2. 策略版本详情页；
3. 策略效果评估页；
4. Agent 调用链可视化页；
5. Wiki 结构优化建议页；
6. 学生长期记忆管理页。

## 18.3 增强版算法

可以引入：

1. 基于时间衰减的掌握度更新；
2. 基于知识图谱的薄弱点传播；
3. 基于错因聚类的错误模式识别；
4. 基于反馈评分的资源生成策略优化；
5. 基于学习路径完成率的推荐策略调整。

示例：

```text
如果学生在“递归调用栈”薄弱，同时“栈”掌握度较低，
则推荐路径应先补“栈的后进先出特性”，再学习“递归调用栈”。
```

# 20. 比赛答辩表述建议

可以这样介绍：

> 智学工坊中的自进化学习智能体不是让 AI 自动修改代码，而是让系统基于学生真实学习过程，持续更新学生画像、薄弱知识点、学习偏好、答疑风格和推荐策略。每一次更新都有行为证据、版本记录、影响范围和回滚机制，因此它是可解释、可控制、可演示的学习策略进化系统。

答辩时重点展示：

1. 学生行为如何被记录；
2. 系统如何发现薄弱点；
3. 系统如何更新画像；
4. 系统如何生成新策略版本；
5. 新策略如何影响后续问答和资源生成；
6. 策略如何查看、解释和回滚。

# 21. 最终结论

智学工坊的自进化学习智能体系统应定位为：

> 面向学生个体的可控学习策略进化系统。

它的核心价值不是“自动化炫技”，而是通过真实学习数据，让系统逐步做到：

1. 更懂学生哪里不会；
2. 更懂学生为什么错；
3. 更懂学生喜欢怎样学；
4. 更懂下一步该推荐什么；
5. 更懂 Wiki 知识空间应该如何补全；
6. 更懂如何生成真正个性化的学习资源。

MVP 阶段应优先跑通：

```text
行为记录 → 薄弱点识别 → 学生画像更新 → 策略版本生成 → 个性化问答变化 → 前端展示进化记录
```

增强版阶段再扩展：

```text
策略效果评估 → Wiki 结构进化 → 多 Agent 可视化 → 风险审核 → 复杂推荐策略优化
```
